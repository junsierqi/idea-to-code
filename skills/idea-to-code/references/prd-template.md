# PRD Template

## Purpose

Use this lightweight PRD when a feature is large enough that coding immediately would create avoidable churn.

## Template

### 1. Goal

- What is being built?
- Why is it being built?

### 2. Users

- Who will use it?
- Which user or operator matters first?

### 3. Main Flows

- Primary happy path
- Key secondary flows
- Important failure or edge flows

### 4. Scope

- In scope
- Out of scope

### 5. Functional Requirements

- Requirement 1
- Requirement 2
- Requirement 3

### 6. Technical Shape

- modules or services touched
- data or protocol changes
- validation approach

### 7. Acceptance

- what must be true for this to count as done
- what evidence will prove it

## Rule

Keep the PRD short and implementation-oriented. It should reduce churn, not become paperwork.
